Typography tokens

Reference for the typography tokens used in this repo. These values are consumed by the compiled Sass tokens and component styles.

Introduction

Typography tokens define the foundational typographic rules of the Percepta design system. They provide a single source of truth for font families, sizes, weights, and line heights used across all components and applications.

These tokens are consumed by the compiled Sass token files and applied consistently through shared components. They are not meant to be used as raw values in applications, but as semantic references that ensure visual and behavioral consistency.

What this page is (and is not)

This page is

  • A reference for available typography tokens
  • A source of truth for typographic foundations
  • A guide to understanding how typography is structured in the system

This page is not

  • A typography style guide
  • A list of CSS rules to copy into applications
  • A replacement for component documentation

Typography decisions should always be applied through the system, not manually.

Base typefaces

Percepta uses two base typefaces, each with a clearly defined role:

  • Noto Sans is used for body text, UI labels, and general interface content.
  • Saans Medium is used for headings and display styles where stronger visual emphasis is required.

This separation helps maintain clarity and hierarchy while keeping the interface neutral and readable.

Why Noto Sans

Noto Sans is a global, Unicode-complete typeface designed to support a wide range of languages and scripts. Using Noto Sans ensures consistent typography across all countries and markets where Giunti Psychometrics operates, without fallback inconsistencies.

Why Saans Medium

Saans Medium is used selectively for headings and display elements to introduce contrast and emphasis without compromising readability. It is not intended for long-form text or dense UI elements.

How these tokens are used

Typography tokens are not applied directly in application code. Instead, they are consumed through:

  • Shared components (e.g. gp-button, gp-input)
  • Semantic typography mixins (e.g. label-control, body, heading)
  • Component-level mappings that combine font size, line height, and spacing

For example:

  • Buttons and form controls use label-control typography to ensure proper vertical rhythm.
  • Body text uses body line height to optimize readability.
  • Headings use heading line height with secondary font family.

This approach allows typography to scale consistently across components without redefining values per use case.
