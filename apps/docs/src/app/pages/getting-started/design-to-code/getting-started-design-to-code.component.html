How Percepta works

Percepta defines how design decisions are translated into production-ready UI. It connects design intent to frontend implementation through tokens, shared components, and documented usage patterns, ensuring consistency across products.

Rather than styling interfaces ad hoc, Percepta establishes a layered system where responsibilities are clearly separated and changes are controlled.

Technical overview

Percepta is built around three core layers:

Design tokens

Design tokens are the single source of truth for:

  • Colors
  • Typography
  • Spacing
  • Border radius
  • Interaction states

Tokens are defined once and compiled into Sass, then consumed by shared components. Application code should never hardcode these values.

Shared components

Shared Angular components (for example buttons, inputs, and icons) are the primary integration point for applications.

Components:

  • Consume tokens
  • Encode interaction behavior
  • Expose a stable API to consuming apps

This ensures that visual and behavioral changes are centralized and predictable.

Documentation

The documentation site is built using the same shared components. What you see in the docs is the real implementation, not mock examples.

Documentation exists to:

  • Explain usage and intent
  • Define interaction contracts
  • Prevent misinterpretation at application level.

Layout responsibility

Percepta separates component behavior from documentation layout.

Shared components (`gp-*`) are layout-agnostic. They do not center themselves, define widths, or control page alignment. Components are designed to behave correctly in any context and must not include layout assumptions.

All layout decisions related to centering, spacing, or presentation live exclusively in the documentation layer.

In this documentation, components may appear centered or constrained for clarity. This is handled by documentation wrappers and is not part of the component implementation.

Components should never manage their own layout.

Technology scope

Percepta is designed to be technology-agnostic at design system level. Design tokens, interaction rules, and component intent are defined independently of any specific framework.

The current implementation uses Angular and PrimeNG as the primary frontend stack, but this represents one concrete implementation, not a limitation of the system.

The same tokens and component definitions can be consumed by other technologies or frameworks by providing an appropriate implementation layer.

PrimeNG integration

PrimeNG provides the structural primitives used by Percepta components.

Percepta:

  • Reuses PrimeNG component structure
  • Overrides styles, tokens, and interaction behavior where needed
  • Avoids forking or reimplementing base components

This approach balances development speed with design control, while staying aligned with Angular best practices.

Governance & contribution

Percepta evolves through controlled changes to tokens and shared components.

  • Proposals can be raised by the frontend team
  • Changes are reviewed before merging
  • Documentation is updated alongside code changes

A breaking change is any modification that:

  • Alters a component API
  • Changes the meaning of an existing token
  • Modifies interaction behavior in a way that affects consuming applications

Breaking changes should be clearly documented and communicated.

Use the project issue tracker to:

  • Report bugs
  • Propose improvements
  • Discuss potential changes before implementation.

Applications should consume Percepta components and tokens directly, rather than redefining UI styles locally.
