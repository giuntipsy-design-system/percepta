Input Text

Single-line text entry for names, search, and short form fields.

Overview Variants State Usage Sizes Interaction mapping Properties

Inputs capture short, structured user input such as names, search queries, or identifiers. They should be clear, predictable, and easy to scan in dense forms.

Use inputs for single-line text entry. For longer content, use a textarea component.

Preview

Variants

Inputs do not use visual variants. Their appearance is controlled exclusively through states to keep forms consistent and predictable.

Usage

Inputs should be clear, concise, and positioned near the content they affect. Use placeholder text to clarify expectations, but rely on labels for context.

Guidelines

  • Always provide a visible label.
  • Use helper text to explain expected formats.
  • Use error messages to explain how to fix invalid input.
  • Avoid disabling inputs unless strictly necessary.

With trailing action

Use trailing actions for quick, contextual controls such as show/hide password or clear input. For password visibility, the icon must represent the action: open eye to show the password, closed eye to hide it. This keeps the affordance focused on the action rather than the current state.

<gp-input leadingIcon="search" placeholder="Search" />

<gp-input label="Password" type="password" [trailingAction]="true">
  <gp-button
    gpInputTrailing
    variant="ghost"
    size="small"
    [iconOnly]="true"
    [icon]="showPassword ? 'eye-closed' : 'eye'"
    [ariaLabel]="showPassword ? 'Hide password' : 'Show password'"
    (clicked)="togglePasswordVisibility()"
  ></gp-button>
</gp-input>
