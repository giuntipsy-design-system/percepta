Button

The primary way users take action in the interface.

Overview Variants State Usage Properties Interaction mapping

Buttons are the primary way to trigger actions: submit, confirm, continue, or run a task.

Variant, label, and state communicate priority and intent. Use buttons for explicit actions and set hierarchy based on impact in the flow.

States include default, hover, active, disabled, and loading. Icons can be added when they improve clarity and accessibility.

Preview

Variants

Button variants define the visual hierarchy and semantic meaning of actions. Each variant communicates the importance and intent of an action, helping users understand what will happen before interacting.

Primary

Used for the main or most important action on a screen. There should typically be only one primary button per view.

Secondary

Used for supporting actions that are important but not the main focus.

Tertiary

Used for low-emphasis actions, often placed alongside content or in secondary contexts.

Text

Used for inline, low-priority actions where minimal emphasis is needed.

Danger

Used for destructive or irreversible actions, such as deleting data or removing content. This variant should be used sparingly and with clear intent.

State

Button states represent how the component responds to user interaction and system conditions. They provide feedback, ensure accessibility, and communicate availability.

Hover

Indicates that the button is interactive when a pointing device is placed over it.

Focus

Shows that the button is focused, supporting keyboard navigation and accessibility.

Active

Represents the pressed state while the user is interacting with the button.

Disabled

Indicates that the action is currently unavailable and cannot be interacted with.

Usage

Buttons are used to trigger tasks, submit information, or confirm decisions. They should be clearly distinguished from navigation links.

If there is no explicit action, use links. If there is a clear action, use buttons.

Guidelines

  • Use the Primary button for the main action on a screen. Limit its use to maintain a clear visual hierarchy.
  • Use Secondary and Tertiary buttons for supporting or low-emphasis actions.
  • Use Text buttons for inline or lightweight actions where minimal emphasis is needed.
  • Use the Danger variant only for destructive or irreversible actions, such as deleting data or removing content.
  • Label buttons with concise, action-oriented text that clearly describes the outcome of the action.
  • Avoid placing too many high-emphasis buttons in the same view, as this reduces clarity and increases cognitive load.
  • Ensure buttons are placed close to the content they affect and follow a logical order within the layout.

Buttons should make the outcome obvious through variant, label, and placement.

Icon-only

Use icon-only buttons for compact actions where the meaning is universally recognized. They work best in toolbars, tight layouts, or repeated actions.

<gp-button variant="secondary" icon="search" [iconOnly]="true" ariaLabel="Search" />
<gp-button variant="tertiary" icon="filter" [iconOnly]="true" ariaLabel="Filter" />

Icon + label

Use icon + label when clarity is critical and the action benefits from both text and a visual cue. This pattern improves recognition in mixed or complex interfaces.

<gp-button label="Search" variant="secondary" icon="search" />
<gp-button label="Filter" variant="tertiary" icon="filter" iconPosition="trailing" />

Sizes

Button sizes ensure consistency across layouts and adapt to different interface densities and contexts. Small: used in compact interfaces, dense layouts, or secondary actions. Medium: the default size, suitable for most use cases across the interface. Large: used for high-emphasis actions or touch-focused interfaces where larger hit areas are required.
